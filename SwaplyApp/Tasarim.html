1. Introduction

This document is an addendum to the Swaply Software Design Document (SDD) submitted on 05.03.2026. As the final phase of the design process, two design problems identified within the existing architecture are addressed by applying formal software design patterns. For each pattern, this document explains: (1) the problem it solves, (2) how it is applied in the Swaply architecture, (3) how the UML class diagrams are updated, and (4) the associated design decisions including quality factors and quality criteria.

2. Design Pattern 1: Observer Pattern

2.1 Problem Identified

In the original design, both the Real-Time Chat (FR4) and the Skill-Based Match Notification (FR3) features relied on ad-hoc Firestore onSnapshot calls scattered directly inside screen components (ChatScreen, MapScreen). This created a tight coupling between the UI layer and the data-listening logic. Specifically:

  • ChatScreen directly contained the Firestore onSnapshot listener for messages.
  • MapScreen independently queried Firestore for nearby matches without any shared notification contract.
  • Adding a new screen that also needs to react to match events would require duplicating listener code.
  • Unit-testing notification behaviour was impossible without rendering full React Native screen components.
This is a classic 'many screens react to the same data event' problem — precisely the scenario the Observer pattern is designed to solve.

2.2 Pattern Overview

  • Pattern Name: Observer
  • GoF Category: Behavioral
  • Intent: Define a one-to-many dependency so that when one object changes state, all dependents are notified and updated automatically.
  • Applied To: FR3 (Match Notifications) and FR4 (Real-Time Chat)

2.3 Application in Swaply Architecture

A new service module, MatchNotificationService.js, is introduced in the /src/services/ layer. It manages a list of subscribers (observers) and owns the single Firestore onSnapshot listener. UI screen components no longer call Firestore directly; instead they register themselves as observers.

Architecture change summary:

  • Before: ChatScreen → Firestore (direct onSnapshot call inside component).
  • After: ChatScreen → MatchNotificationService → Firestore. ChatScreen acts as an observer and subscribes to the service via a standard Javascript event interface.
  • The same MatchNotificationService also notifies FCM (push notification dispatch) when a new match document is created, keeping all notification logic centralized.
  • MapScreen acts as an observer to react to new nearby matches without owning any Firestore listener.

2.4 Updated UML Class Diagram

Relationship: MatchNotificationService acts as the Subject. ChatScreenObserver and MapScreenObserver implement the IMatchObserver behavior.

2.5 Design Decision

Why Observer was chosen:
The Swaply application has multiple screens that must react to the same Firestore events. Without a formal pattern, each screen maintains its own listener, leading to resource waste and code duplication. The Observer pattern provides a single listener that fans out to registered screens, reducing Firestore read costs and isolating the change-detection logic from the UI.

Architectural location:
The Subject (MatchNotificationService.js) lives in /src/services/. Concrete observer implementations are co-located with their respective screen components in /src/screens/ to keep screen-specific reaction logic close to the view.

Quality FactorMetric / CriteriaTarget Value
MaintainabilityAdding a new screen that reacts to match events requires only subscribing the observer — zero changes to existing classes.Cyclomatic complexity per function < 10 (ESLint).
PerformanceSingle Firestore onSnapshot listener shared by all observers instead of N separate listeners.≤ 1 active snapshot listener per collection at runtime.
TestabilityService can be unit-tested by injecting a mock observer function without any React Native rendering.100% of service methods covered by Jest unit tests.

3. Design Pattern 2: Repository Pattern

3.1 Problem Identified

In the original design, Firebase/Firestore API calls (getUserById, findMatches, submitRating, deleteUser) were made directly inside screen components and isolated service files without a consistent abstraction layer. This caused:

  • Screen components (e.g., ProfileScreen, DiscoverScreen) indirectly depending on the concrete cloud provider details.
  • Duplicated Firestore query logic — the same 'find users by skill overlap' query could appear in different parts of the application.
  • No clear contract for what data operations the application requires, making it difficult to mock the database during testing.
  • FR10 (Account Deletion) required touching multiple files to gather all user-related collections, lacking a single origin for 'all user data operations'.
This violates the Dependency Inversion Principle and makes the data layer difficult to test or change independently of the UI.

3.2 Pattern Overview

  • Pattern Name: Repository
  • Category: Architectural / Structural Data Access
  • Intent: Encapsulate all data access logic behind a centralized layer/interface, decoupling domain/business logic from the data source provider.
  • Applied To: FR1, FR2, FR7, FR8, FR9, FR10 — all features requiring user data read/write operations.

3.3 Application in Swaply Architecture

A central javascript module UserRepository.js is introduced containing all user-related data operations, formally replacing scattered service calls. Screen components depend only on the exported UserRepository functions — never on firebaseConfig.js imports directly.

Architecture change summary:

  • Before: ProfileScreen → import { db } from '../services/firebaseConfig' → db.collection('users').doc(uid).get().
  • After: ProfileScreen → UserRepository.getUserById(uid).
  • The /src/services/ folder structurally isolates database interaction. firebaseConfig.js is wrapped by UserRepository.js.
  • FR10 deleteUser(uid) now calls a single repository method that atomically deletes the user document, all match documents, and all chat subcollections.

3.4 UML Class Diagram

Relationship: FirestoreUserRepository fulfills the data access contract. Application controllers and services interact with data strictly through this centralized API (Dependency Inversion).

3.5 Design Decision

Why Repository was chosen:
The Swaply project uses Firebase as its sole backend, but tying screen logic tightly to the Firestore SDK restricts testing capabilities and violates Separation of Concerns. By placing all data access behind UserRepository.js, the migration/refactoring surface is reduced to a single module.
Additionally, the Repository pattern enables Jest mock implementations to be injected during unit tests, removing the need to connect to a live Firestore instance for every screen test — directly supporting the QA Plan's goal of automated unit testing.

Architectural location:
The implementation resides in /src/services/UserRepository.js. Firebase instances are privately scoped within this module and never exposed to the UI screens.

Quality FactorMetric / CriteriaTarget Value
MaintainabilityAll Firestore-specific code is isolated to Repository. Replacing Firestore requires modifying only one file.0 direct Firebase SDK imports in /src/screens/.
TestabilityRepository methods can be mocked in Jest. FR10 & FR8 can be fully tested without a live DB connection.> 80% code coverage on all repository methods (Jest).
Security (NFR2)Firebase Security Rules are validated at a single layer. The repository is the only place allowed to write to users.0 unauthorized write paths outside the Repository layer.
